# SkyQuery – платформа для поиска и бронирования авиабилетов.

Приложение состоит из двух микросервисов – auth и flight. Первый отвечает за авторизацию пользователей, а второй за основную бизнес логику. Оба микросервиса подняты с помощью docker compose и каждый из них имеет свою базу данных, поднятую в отдельном контейнере. Все контейнеры находятся в одной сети докера, причем снаружи доступен только flight service. 

## Auth service
Приложение реализует паттерн централизованной аутентификации, то есть решение о том, доступен ли определенный ресурс конкретному пользователю, определяется в одном месте, а именно решение принимает auth service. 

Аутентификация построена на JWT паре токенов access + refresh, где оба токена передаются в cookie с соответствующими названиями и samesite=strict для защиты от csrf атак. Refresh токены хранятся в redis для того, чтобы быстро получать к ним доступ и инвалидировать при надобности.

Любой входящий запрос сначала попадает в flight service, в котором содержится прокси-контроллер auth сервиса, содержащий все те же самые ендпоинты, что и auth сервис, за исключением /api/v1/auth/validate, который необходим для валидации входящего запроса. После того, как запрос попадает в flight сервис, он делает синхронный запрос на этот самый /api/v1/auth/validate для того, чтобы получить информацию о том, доступен ли определенный ресурс и метод пользователю, который сделал запрос. Информация о том, какими правами необходимо обладать для доступа к ресурсу статично задана в yaml конфигурации auth сервиса для того, чтобы не перекомпилировать сервис при изменении правил доступа.

## Flight service
Основной сервис, содержащий бизнес логику поиска и бронирования авиабилетов. Логически поделен на 4 домена: booking, flight, pricing, reference data.

Для просмотра списка авиаперелетов по базовым выбранным фильтрам(город вылета, город прилета, время вылета, время прилета(необязательный параметр), класс места, тип и количество пассажиров), конкретной информации об авиаперелете, списке авиаперелетов с динамическими фильтрами(то есть информация о том, какие фильтры доступны конкретно для этого базового поискового запроса отдается клиенту, чтобы не показывать ему те фильтры, при примении которых не будет результатов), списке билетов, конкретной информации о билете, то есть для тяжелых запросов на поиск данных используется JOOQ – библиотека, типобезопасный динамический конструктор запросов на SQL, позволяющий избежать магии ORM, когда непонятно, какой и сколько запросов ушло в БД, но при этом позволяющий писать запросы на языке Java, то есть не слишком низкоуровневый, как, например, инструмент JDBC для доступа к данным. В некоторых репозиториях использовался JPQL и CriteriaBuilder(Route модель и все, что с ней связано, в домене Reference Data)

Для всех остальных запросов на изменение данных использовался классчиеский Spring Data JPA, что позволило реализовать паттерн CQRS-lite, который заключается в том, чтобы разделить модель чтения данных и модель изменения данных.

### Booking
Домен, отвечающий за бронирования, просмотра билетов и отмены бронирования. 

### Flight
Домен, отвечающий за сами авиаперелеты. Для владельцев авиакомпаний(то есть админов) доступно добавление, обновление и удаление конкретных перелетов.

### Pricing
Домен, отвечающий за цены на авиаперелеты. Доступен только для админов, они могут: добавлять или удалять тарифы, изменять цены на конкретные тарифы для всех категорий пассажаиров в связи со спросом или близким к вылету временем, изменять цены на тарифы для конкретных типов пользователей или конкретные классы билетов(эконом, бизнес, первый класс и т.п.).

### Reference data
Домен, отвечающий за справочную информацию, включает в себя информацию о: странах, городах, местах в самолетах(только о том, какие в принципе есть места в конкретном типе самолета, информация о статусах мест содержится в домене Flight), аэропортах, типах самолетов, конкретных самолетах, авикомпаниях и маршрутах. Доступно только для админов.
